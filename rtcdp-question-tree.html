Q

"What does 'real-time' actually mean in practice?"

Technical

Recommended Response

Real-time means customer profiles update and segments are evaluated in milliseconds, not hours or days. When someone lands on your website, RT-CDP instantly pulls their unified profile, evaluates them against active segments, and makes activation decisions — all within a single HTTP request. So a customer who just browsed a product can see a personalized recommendation within 500ms. Compare that to batch CDPs where audiences are built once a day or once a week. For time-sensitive moments — a customer at checkout, or a visitor abandoning your site — those milliseconds make the difference between conversion and lost revenue.

Q

"How does Real-Time CDP build unified customer profiles?"

Technical

Recommended Response

RT-CDP ingests data from every source you have — websites, apps, email clicks, CRM systems, call centers, loyalty platforms, offline purchases — and stitches them together using Identity Service. It resolves multiple identifiers (email, CRM ID, phone, device IDs, loyalty numbers) into a single identity graph, then consolidates all the behaviors, attributes, and interactions associated with that person into one unified profile. As new data arrives, the profile updates in real time. The result is a 360-degree view of each customer that's always current and available for activation across any channel.

Q

"How does it handle known vs. anonymous visitors?"

Technical

Recommended Response

RT-CDP handles both seamlessly. For known customers (logged in, identifiable via email or CRM), it builds a complete unified profile with all their history and attributes. For anonymous visitors, it still creates a profile (using device IDs, cookies, IP addresses) so you can personalize their experience and track their journey. When an anonymous visitor eventually identifies themselves — by logging in or converting — RT-CDP instantly merges their anonymous profile with their known profile, so no behavioral history is lost. This is critical because most of your revenue comes from converting anonymous visitors into known customers, and you need that complete journey to personalize effectively.

Q

"What about data quality — garbage in, garbage out?"

Objection

Recommended Response

That's a legitimate concern, and the answer is that RT-CDP has built-in data governance and quality tools. The DULE (Data Usage Labeling and Enforcement) framework lets you tag data quality levels, flag suspicious data, and enforce standards. You can set up validation rules so bad data is quarantined before it hits profiles. You can also run Data Hygiene and Standardization jobs to clean and deduplicate data as it ingests. The key is treating data governance as part of implementation, not an afterthought. If your source data is garbage, RT-CDP won't magically fix it — but it gives you visibility and controls to prevent bad data from poisoning downstream activations.

Q

"How does Real-Time CDP handle audience segmentation?"

Discovery

Recommended Response

Real-Time CDP includes Segment Builder — a visual, no-code interface where you drag and drop conditions to create audiences. You can segment by attributes (age, location, loyalty tier), behaviors (pages visited, products viewed, email opens), or temporal logic (users who did X in the last 30 days). Segments evaluate in real time, so as soon as a customer meets the criteria, they're added to the audience. You can also use AI-driven features like predictive audiences — lookalike audiences based on your high-value customers, or propensity-to-churn segments that automatically identify at-risk users. And if you need advanced logic, you can write in SQL or use Audience Composition to combine and refine segments.

Q

"We're worried about data privacy and consent with a CDP."

Objection

Recommended Response

This is actually a strength of RT-CDP, not a weakness. Privacy compliance is built in — not something you bolt on later. The DULE framework lets you tag data with restrictions (e.g., "this is PII," "this can't be used for paid social"), and those labels are automatically enforced across every destination. You set consent policies at the segment level — if a customer hasn't consented to email, they'll never be added to an email segment, even if they match the criteria. And RT-CDP integrates with consent management platforms (Osano, OneTrust, AdobeCMP) to sync consent decisions in real time. You can also handle GDPR right-to-be-forgotten requests by triggering bulk deletion across all your destinations. Many organizations find that RT-CDP actually improves their compliance posture vs. running a fragmented stack.

Q

"How does it handle GDPR and CCPA compliance?"

Technical

Recommended Response

Real-Time CDP has native GDPR and CCPA workflows. For GDPR's right to be forgotten, you can delete a customer record and RT-CDP automatically triggers deletion across all connected destinations — Marketo, email platforms, ad tech, data warehouses. For CCPA's right to access, you can export a customer's complete data profile. You can also set up automated data retention policies so data expires after a specified period. The Privacy Service dashboard gives you visibility into all requests and their status. There's also a Privacy Shield feature that encrypts sensitive PII fields. None of this is foolproof — you still need legal review and proper processes — but RT-CDP gives you the technical infrastructure to be compliant.

Q

"What about consent management across channels?"

Technical

Recommended Response

RT-CDP can be the system of record for consent. You capture consent once (via your website, app, email preference center, or partner CMP), store it in unified profiles with timestamps, and enforce it consistently across email, SMS, push, paid social, and data warehouse activation. So if a customer opts out of marketing emails, that decision automatically prevents them from being added to email segments — even if they match the targeting criteria. You can also do granular consent per purpose (email marketing, analytics, ads) or per channel. The key architectural decision is whether RT-CDP is your consent source of truth, or if it syncs with an external CMP. Either way, centralized enforcement is infinitely better than each tool managing consent independently.

"How does Real-Time CDP compare to Segment, Treasure Data, or Tealium?"

Competitive

Recommended Response

Real-Time CDP's key differentiators are: (1) True real-time evaluation (milliseconds, not batches); (2) Native integration with 5 other Adobe products, so your CDP data feeds analytics (CJA), personalization (Target), orchestration (Journey Optimizer), and more without custom integrations; (3) AI/ML capabilities built in — predictive audiences, lookalike modeling, and propensity scoring via Adobe Sensei; (4) DULE privacy framework with automatic enforcement across destinations; and (5) Segment Match for second-party data collaboration. Segment is strong for tag management and event collection but weak on activation and lacks advanced AI. Treasure Data and Tealium have longer implementation timelines and don't have the same native application ecosystem. If you're already in Adobe (Analytics, Target, Campaign), RT-CDP is a natural fit. If you're building a best-of-breed stack, it's still competitive on speed and AI.
